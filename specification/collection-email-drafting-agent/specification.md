## MCP Tool Wiring â€” Dunning History (Existing MCP Server)

The dunning history MCP server already exists externally. Tool schemas are saved at:
- `specification/collection-email-drafting-agent/mcp-specs/mcp-spec-dunning-history-list_dunning_runs.json`
- `specification/collection-email-drafting-agent/mcp-specs/mcp-spec-dunning-history-list_dunned_items.json`

MCP Server ORD ID: `sap.mcpbuilder:apiResource:contract_accounting_dunning_history_mcp_demo:v1`

Available tools (all read-only):
- `get_metadata_ca_dunning` â€” retrieve OData service metadata; call before other dunning tools
- `list_dunning_runs` â€” list dunning run headers by BusinessPartner (filter: `BusinessPartner eq '<id>'`); key fields: `CAMassRunDate`, `CAMassRunID`, `BusinessPartner`, `ContractAccount`, `CADunningLevel`, `CADunningBalance`, `TransactionCurrency`, `CANextDunningDate`
- `list_dunned_items` â€” list individual dunned items with amounts and due dates (filter: `BusinessPartner eq '<id>'`); key fields: `CADocumentNumber`, `CADunningAmount`, `TransactionCurrency`, `CANetDueDate`, `CADunningLevel`
- `get_dunning_run_by_key` â€” retrieve a specific dunning run by key (requires `camassrundate`, `camassrunid`, `businesspartner`, `contractaccount`, `cadunningcounter`)

- [ ] Wire dunning MCP server tools in `agent.py` using `get_mcp_tools()` from `mcp_tools.py` â€” NEVER create direct HTTP clients
- [ ] Add dunning MCP server to `asset.yaml` under `requires`:
  ```yaml
  requires:
    - name: contract-accounting-dunning-history-mcp-demo
      kind: mcp-server
      ordId: sap.mcpbuilder:apiResource:contract_accounting_dunning_history_mcp_demo:v1
  ```

## MCP Translation File â€” Dispute Management API (No Existing MCP Server)

The Dispute Management OData API has no existing MCP server. An MCP translation file must be generated.

API ORD ID: `sap.s4:apiResource:CE_API_DISPUTE_MANAGE_0001:v1`
API spec: `specification/collection-email-drafting-agent/api-specs/dispute-manage.json`

Key endpoints to expose via translation:
- `GET /DebtCollectionsDispute` â€” list disputes; filter by `BusinessPartner`; key response fields: `DCoDisputeUUID`, `DebtCollectionsDispute`, `BusinessPartner`, `DebtCollectionsCompanyCode`, `DCoDisputeProcessStep`, `DCoDisputeProcessStepStatus`, `DCoDisputePriority`, `DCoDisputeCurrency`, `DCoDisputeProcessor`, `DCoDisputePlannedCloseDate`
- `GET /DebtCollectionsDispute/{DCoDisputeUUID}` â€” retrieve single dispute by UUID

## Agent System Prompt

- [ ] Implement `@prompt_section` system prompt in `app/agent.py` that:
  - Defines the agent role: "You are a collections assistant that helps AR specialists draft personalized collection emails"
  - Instructs: always call `get_metadata_ca_dunning` before any dunning tool call on first use per session
  - Instructs: always set `top` to a maximum of 100 on every tool call that accepts it; inform the user when this limit is applied
  - Instructs: NEVER fabricate or estimate financial figures â€” all amounts, dates, and invoice numbers must come verbatim from tool call responses
  - Instructs: when the user provides a customer/business partner identifier, retrieve all available context (dunning runs, dunned items, disputes) before generating the draft
  - Instructs: present the complete draft and explicitly ask the specialist to review, edit if needed, and confirm approval before sending
  - Instructs: NEVER send or confirm dispatch of any email without an explicit approval message from the specialist in the current session
  - Instructs: the tone of all emails must be professional, firm, and respectful â€” avoid aggressive or overly casual language
- [ ] Set `@agent_config` temperature to `0.3` (low for factual/precise email drafting)

## Customer Context Assembly (REQ-01)

- [ ] Implement context assembly logic that, given a customer identifier (BusinessPartner ID or name), calls:
  1. `list_dunning_runs` filtered by `BusinessPartner` â€” retrieve dunning history with levels and balances
  2. `list_dunned_items` filtered by `BusinessPartner` â€” retrieve individual overdue items with amounts and due dates
  3. List disputes via dispute MCP tool filtered by `BusinessPartner` â€” retrieve open dispute cases
- [ ] Assemble results into a structured context object with: overdue items, dunning history summary, active disputes
- [ ] Instrument milestone M1 (Overdue Receivables Lookup):
  - On success: `logger.info("M1.achieved: overdue open items retrieved for customer %s, %d items found", customer_id, count)`
  - On failure: `logger.warning("M1.missed: overdue open items retrieval failed or returned no results for customer %s", customer_id)`
- [ ] Instrument milestone M2 (Context Assembly):
  - On success: `logger.info("M2.achieved: full context assembled for customer %s; dunning entries: %d, disputes: %d", customer_id, dunning_count, dispute_count)`
  - On failure: `logger.warning("M2.missed: context assembly incomplete for customer %s; missing data sources: %s", customer_id, missing)`

## Email Draft Generation (REQ-02)

- [ ] Implement email draft generation using the LLM (via SAP Generative AI Hub / LiteLLM) with the assembled customer context
- [ ] The generated draft must include:
  - Subject line referencing the customer name and overdue amount
  - Professional greeting
  - Clear statement of overdue invoice details (invoice numbers, amounts, original due dates) sourced from tool responses
  - Explicit payment request with a deadline date
  - Reference to dunning level / escalation status if applicable
  - Professional closing with contact details placeholder
- [ ] If disputes are active, include a note acknowledging the dispute and requesting resolution or payment of undisputed amounts
- [ ] Instrument milestone M3 (Email Draft Generation):
  - On success: `logger.info("M3.achieved: email draft generated for customer %s", customer_id)`
  - On failure: `logger.warning("M3.missed: email draft generation failed for customer %s; reason: %s", customer_id, reason)`

## Specialist Review and Approval Gate (REQ-03)

- [ ] After presenting the draft, the agent must explicitly ask: "Please review the draft above. Reply 'approve' to confirm sending, or provide your edits."
- [ ] The agent must not proceed to dispatch confirmation unless the specialist sends an explicit approval
- [ ] If the specialist edits the draft inline, regenerate or acknowledge the updated version and re-confirm approval
- [ ] Instrument milestone M4 (Specialist Review):
  - On approval: `logger.info("M4.achieved: draft approved by specialist for customer %s", customer_id)`
  - On session end without approval: `logger.warning("M4.missed: specialist did not approve draft for customer %s; session ended without confirmation", customer_id)`

## Email Dispatch Confirmation (REQ-03 / M5)
